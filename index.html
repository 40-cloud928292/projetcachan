PROJET ROBOTIQUE CACHAN

Configuration et optimisation de la plateforme Pololu 3Pi+

GEII Vélizy | Concours de Robotique de Cachan

Présentation du Projet

Ce site présente les travaux de développement réalisés par notre équipe de trois étudiants en Génie Électrique et Informatique Industrielle (GEII) de l'IUT de Vélizy dans le cadre du concours de Cachan. Ce concours évalue les performances d'architectures robotiques autonomes sur des parcours spécifiques.

Notre approche repose sur l'exploitation approfondie, l'étalonnage et la programmation logicielle d'un robot mobile pré-assemblé Pololu 3Pi+, configuré pour maximiser la vitesse de traitement des données et la précision des trajectoires.

ARCHITECTURE DE LA PLATEFORME

Spécifications techniques du robot d'origine Pololu 3Pi+

Pololu 3Pi+ Diagram
Châssis et Intégration
Structure mobile pré-assemblée compacte conçue pour les évolutions en milieu contraignant, associant légèreté et faible inertia de rotation.
Groupe Motopropulseur
Deux moteurs à courant continu couplés à des réducteurs mécaniques assurant un couple constant, une accélération linéaire et une rétroaction par encodeurs.
Réseau de Capteurs Infrarouges
Barrette frontale de 5 capteurs de réflectance infrarouge dédiée à la discrimination fine des lignes de guidage au sol et au repérage des repères topographiques.
Unité de Traitement Électronique
Microcontrôleur cible compatible AVR cadencé à une fréquence d'horloge de 20 MHz, optimisé pour exécuter la boucle d'asservissement principale sans latence.
Sous-système d'Alimentation
Régulation de tension embarquée alimentée par accumulateurs, dimensionnée pour maintenir une tension nominale stable et garantir une autonomie adaptée à la durée du concours.

Parcours 3 - Détection de Tirés (Dashes)

Algorithme de suivi de ligne avec détection de 4 tirés transversaux isolés (repères noirs perpendiculaires à la trajectoire). Le robot s'arrête immédiatement après franchissement du 4ème tiré. Utilise un filtre anti-rebond pour éviter les faux comptages et une régulation proportionnelle simple (Kp = 0.06).

#include <Pololu3piPlus32U4.h>
using namespace Pololu3piPlus32U4;

LineSensors sensors;
Motors motors;

uint16_t s[5];
int baseSpeed = 110;
float Kp = 0.06;
int lastError = 0;

int noLineCount = 0;
const int NO_LINE_THRESHOLD = 25;
const int LINE_DETECT_MIN = 200;

int crossingCount = 0;
const int MAX_CROSSINGS = 4;
bool onDash = false;
bool waitingToStop = false;
int debounceCount = 0;
const int DEBOUNCE_LOOPS = 80;

#define DASH_SENSOR    s[4]
#define DASH_NEIGHBOR  s[3]
const int DASH_THRESHOLD = 500;

void setup()
{
  delay(1000);
  for (int i = 0; i < 80; i++)
  {
    motors.setSpeeds((i < 40) ? 60 : -60, (i < 40) ? -60 : 60);
    sensors.calibrate();
  }
  motors.setSpeeds(0, 0);
}

void loop()
{
  sensors.readCalibrated(s);

  int maxVal = 0;
  for (int i = 0; i < 5; i++)
    if (s[i] > maxVal) maxVal = s[i];

  if (maxVal < LINE_DETECT_MIN)
  {
    noLineCount++;
    if (noLineCount >= NO_LINE_THRESHOLD)
    {
      motors.setSpeeds(0, 0);
      while (1);
    }
  }
  else noLineCount = 0;

  if (debounceCount > 0)
  {
    debounceCount--;
  }
  else
  {
    bool dashAlone = (DASH_SENSOR > DASH_THRESHOLD) && (DASH_NEIGHBOR < DASH_THRESHOLD);

    if (dashAlone && !onDash)
    {
      onDash = true;
      crossingCount++;
      debounceCount = DEBOUNCE_LOOPS;

      if (crossingCount >= MAX_CROSSINGS)
        waitingToStop = true;
    }

    if (!dashAlone && onDash)
    {
      onDash = false;

      if (waitingToStop)
      {
        motors.setSpeeds(0, 0);
        while (1);
      }
    }
  }

  int error = (s[4] + s[3]) - (s[0] + s[1]);
  if (s[2] > (s[1] + s[3]) / 2)
    error = (s[3] - s[1]);
  if (abs(error - lastError) > 1500)
    error = lastError;

  lastError = error;

  int correction = error * Kp;
  int leftSpeed  = baseSpeed + correction;
  int rightSpeed = baseSpeed - correction;
  leftSpeed  = constrain(leftSpeed, -150, 200);
  rightSpeed = constrain(rightSpeed, -150, 200);
  motors.setSpeeds(leftSpeed, rightSpeed);
}

⚠️ Rapport d'Essai - Problématique d'Aléas (Instabilité Algorithmique)

« Ça fonctionne une fois sur deux... c'est aléatoire... il n'y a rien qui change dans le code. »
Diagnostic de l'équipe : La boucle d'asservissement subit des sautes de comportement cinématique (ralentissements inexpliqués et pertes intermittentes du suivi de ligne). L'étalonnage de la barrette de réflectance infrarouge est fortement sujet aux variations de la luminosité ambiante du laboratoire et aux micro-reliefs du support. Une optimisation du profil logiciel et une stabilisation des seuils de détection sont impératives avant le passage devant le jury de Cachan.
